## 2. Текущая фаза

**Фаза 0 — каркас моно-репо.** Готово: структура папок, docker-compose, FastAPI-скелет с `/healthz`, три фронта с заглушечными «Hello»-страницами, дизайн-система Делифон (токены в `packages/shared-types`).

Бизнес-логики в коде нет. Следующие фазы (§19.1):
1. Фаза 1 — модели БД, auth (клиент SMS, партнёр, админ+2FA), базовые CRUD
2. Фаза 2 — клиентское PWA (KYC, карта, бронирование, аренда)
3. Фаза 3 — партнёрский кабинет (мастера выдачи/приёма)
4. Фаза 4 — инциденты и финальная админка
5. Фаза 5 — уведомления, чат, доводка

## 3. Правила разработки

### Архитектура
- API разделяется на `/api/v1/{client,partner,admin,public,webhooks}` (§3.3). Не смешивай — каждый фронт ходит только в свой префикс.
- Auth: клиент — JWT по SMS; партнёр — JWT email+пароль; админ — JWT + обязательный 2FA TOTP (§7.2).
- БД: SQLAlchemy 2.0 async, таблицы и поля из §13 (snake_case, множественное число, UUID v4, `created_at`/`updated_at`). Менять схему — только через Alembic (`make migration name=...`).
- Event-driven задачи (автосписания, напоминания) — через Celery в `app/workers/` (§3.2).
- Идемпотентность платежей — обязательна. Любая операция может быть повторена без двойного списания (§3.2).
- Audit log — по умолчанию для всех действий операторов/админов (§3.2, таблица `audit_log` в §13).

### Дизайн-система
- Все цвета, типографика, радиусы, тени — только из `@deliphone/shared-types/tokens` (`DelifonTokens`). Не хардкодь hex в компонентах. Если не хватает токена — добавляй его в `tokens.ts` и используй во всех фронтах.
- Tailwind-классы: `bg-accent`, `bg-ink-900`, `text-ink-500`, `rounded-full`, `shadow-elev-2` и т. д. — уже настроены.
- Ant Design: тема в `apps/admin/src/theme/antdTheme.ts`. Хочешь добавить новый компонент-кастомизацию — правь там.

### Код
- Не добавляй абстракции «на будущее» — реализуй ровно то, что просит задача или спека. Не делай feature-flags и backwards-compatibility shims, если можно просто изменить код.
- Не пиши комментарии «что делает код» — они устаревают. Комментарий нужен только тогда, когда объясняет **почему** что-то сделано именно так (скрытый инвариант, обход бага, требование 152-ФЗ и т. п.).
- Не создавай `.md`-файлы «по ходу» (планы, решения, заметки) без явной просьбы пользователя.
- Безопасность: не вноси уязвимости OWASP Top 10. Секреты — только из `.env`, никогда не коммить.

## 4. Коммиты и PR

**Conventional Commits** строго:

```
<type>(<scope>): <subject>

<body, при необходимости>
```

Типы: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `perf`, `style`, `ci`, `build`.

Scope — доменная область или приложение: `backend`, `client`, `partner`, `admin`, `infra`, `workspace`, `db`, `auth`, `kyc`, `payments`, `incidents`…

Правила:
- Коммиты атомарные — один коммит = одна логическая единица работы.
- Не смешивай рефакторинг с новой фичей.
- Сообщение на английском, тело — при необходимости, более подробное описание «зачем».
- Никогда не используй `--no-verify` и не пуш-форсируй в `main`.

Примеры:
- `feat(backend): add /api/v1/client/auth/phone endpoint`
- `feat(client): implement KYC passport upload step`
- `fix(backend): idempotent payment retry for daily_charge`
- `refactor(partner): extract device-issue wizard into steps`

## 7. Полезные напоминания
- Клиенту обращаемся на «ты» (§1.8).
- Хостинг и БД — только в РФ (242-ФЗ, §15).
- Паспортные данные — AES-256 в БД, фото — доступ через журнал (§15.1).
- Холды ЮKassa — до 180 дней, категория «аренда/прокат» (§1.7, §9).
- ЮKassa webhook'и — с проверкой HMAC (§14.4).

Если что-то в задаче противоречит спеке — останавливайся и уточняй у пользователя.
